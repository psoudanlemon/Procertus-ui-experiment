## 01. The hierarchy of intent

Every interactive overlay in the PROCERTUS system has a permanent z-address.
This is not a guideline. It is a contract. A dropdown will always sit at the
same altitude as a popover, a modal will always eclipse a sheet, and a toast
will always float above everything else. The user never has to wonder which
layer will win.

    <p className="text-sm text-muted-foreground m-0" style={{ lineHeight: 1.6 }}>
      The page itself is the ground floor: cards, buttons, containers. Above it, a
      promoted element lifts just enough to distinguish itself. Ephemeral hints like
      tooltips float higher still. Interactive menus, sheets, and popovers occupy the
      next altitude. Modals and command palettes demand full attention above those.
      System notifications sit above everything for alerts that must never be occluded.
    </p>
  </div>
  <div className="rounded-xl border border-border bg-card p-6">

    <p className="text-sm text-muted-foreground m-0" style={{ lineHeight: 1.6 }}>
      A compliance officer reviewing an ISO 27001 audit trail may have a data table,
      a side sheet, a command palette, and a toast on screen simultaneously. Without a
      fixed hierarchy, the brain wastes energy figuring out what is in front of what.
      The tier system removes that cognitive tax entirely: each element's altitude is
      predetermined, so spatial reasoning becomes unconscious.
    </p>
  </div>
</div>

## 02. The physics of depth

Shadows in this system are a finite resource. Every shadow token represents a
measured amount of visual weight. Spend too much and the interface feels heavy,
muddy, and anxious. Spend too little and everything looks flat and disconnected.
The budget is strict: five neutral shadow values, one backdrop blur, and zero
colored glows in the elevation stack.

## 03. Adaptive ergonomics

On a desktop, a dropdown floats beside its trigger: small, precise, out of the way.
On a phone, that same floating rectangle is too small to tap, too far from the thumb,
and too easy to accidentally dismiss. The solution is not to shrink the desktop pattern.
It is to replace it entirely with a pattern built for the hand.

<div className="flex flex-col mt-6" style={{ gap: '24px' }}>

<div className="rounded-xl border-2 border-brand-accent-200 dark:border-brand-accent-800 bg-card p-6">
  <span className="font-semibold text-foreground block" style={{ fontSize: '1.5rem' }}>Swap, do not shrink</span>
  <span className="uppercase block mb-3" style={{ fontSize: '12px', fontWeight: 600, letterSpacing: '0.1em', color: 'var(--muted-foreground)', marginTop: '8px' }}>The doctrine</span>
  <p className="text-sm text-muted-foreground m-0" style={{ lineHeight: 1.6 }}>
    A responsive dropdown is not a smaller dropdown. It is a drawer. A responsive
    modal is not a narrower modal. It is a drawer anchored to the thumb zone. A
    responsive command palette is not a squished search box. It is a full-page sheet
    with room to type, scan, and select. Every mobile substitution is a completely
    different component chosen for ergonomic fitness, not a CSS media query that
    adjusts padding.
  </p>
</div>

<div className="grid grid-cols-1 md:grid-cols-2 gap-6">
  <div className="rounded-xl border border-border bg-card p-6">
    <span className="font-semibold text-foreground block" style={{ fontSize: '1.5rem' }}>Linear content</span>
    <span className="uppercase block mb-3" style={{ fontSize: '12px', fontWeight: 600, letterSpacing: '0.1em', color: 'var(--muted-foreground)', marginTop: '8px' }}>Drawer</span>
    <p className="text-sm text-muted-foreground m-0" style={{ lineHeight: 1.6 }}>
      Flat lists of actions, simple selections, confirmation prompts: anything the user
      can resolve with a single tap. These become drawers that slide up from the bottom
      edge with <strong>44px</strong> minimum touch targets. The content is linear, so
      the container is linear.
    </p>
  </div>
  <div className="rounded-xl border border-border bg-card p-6">
    <span className="font-semibold text-foreground block" style={{ fontSize: '1.5rem' }}>Depth content</span>
    <span className="uppercase block mb-3" style={{ fontSize: '12px', fontWeight: 600, letterSpacing: '0.1em', color: 'var(--muted-foreground)', marginTop: '8px' }}>Full-page sheet</span>
    <p className="text-sm text-muted-foreground m-0" style={{ lineHeight: 1.6 }}>
      Nested menus, searchable lists, rich content panels: anything that requires
      navigation or filtering. These become full-page sheets that slide in from the
      right with a back button and room for a search field. The content has depth, so
      the container provides depth.
    </p>
  </div>
</div>

<div className="rounded-xl border-2 border-sys-warning-300 dark:border-sys-warning-700 bg-sys-warning-50 dark:bg-sys-warning-950 p-6" style={{ marginTop: '24px' }}>

## 04. System orchestration

Depth is not static. It is revealed through motion. When a sheet slides in from the
right, the user understands it is a layer on top of the page. When a modal fades in
with a subtle scale, the user understands it is demanding attention. When a toast
appears at the bottom corner, the user understands it is a notification that will leave
on its own. Motion is the verb that makes the elevation system legible.

<div className="flex flex-col mt-6" style={{ gap: '24px' }}>

<div className="grid grid-cols-1 md:grid-cols-2 gap-6">
  <div className="rounded-xl border-2 border-brand-primary-200 dark:border-brand-primary-800 bg-card p-6">
    <span className="font-semibold text-foreground block" style={{ fontSize: '1.5rem' }}>Instance swap</span>
    <span className="uppercase block mb-3" style={{ fontSize: '12px', fontWeight: 600, letterSpacing: '0.1em', color: 'var(--muted-foreground)', marginTop: '8px' }}>The rule</span>
    <p className="text-sm text-muted-foreground m-0" style={{ lineHeight: 1.6 }}>
    At the highest attention tier, only one overlay may be active at a time. Opening a
      command palette while a modal is visible must close the modal first. This is not
      a technical limitation. It is a design constraint: the user's attention is a
      resource, and two full-attention surfaces competing for it is a failure of
      orchestration. The new overlay replaces the existing one.
    </p>
  </div>
  <div className="rounded-xl border border-border bg-card p-6">
    <span className="font-semibold text-foreground block" style={{ fontSize: '1.5rem' }}>Motion as syntax</span>
    <span className="uppercase block mb-3" style={{ fontSize: '12px', fontWeight: 600, letterSpacing: '0.1em', color: 'var(--muted-foreground)', marginTop: '8px' }}>How depth speaks</span>
    <p className="text-sm text-muted-foreground m-0" style={{ lineHeight: 1.6 }}>
      Each tier enters and exits with a motion that communicates its spatial position.
      Sheets slide from the edge they are anchored to. Modals scale up from the center.
      Dropdowns and popovers fade with a subtle zoom. Toasts slide in from the screen
      boundary. The user never has to read a label to understand what kind of overlay
      appeared: the motion told them.
    </p>
  </div>
</div>

<div className="rounded-xl border border-border bg-card p-6">
  <span className="font-semibold text-foreground block" style={{ fontSize: '1.5rem' }}>Stacking sheets</span>
  <span className="uppercase block mb-3" style={{ fontSize: '12px', fontWeight: 600, letterSpacing: '0.1em', color: 'var(--muted-foreground)', marginTop: '8px' }}>The exception</span>
  <p className="text-sm text-muted-foreground m-0" style={{ lineHeight: 1.6 }}>
    Sheets are the one element that may stack: opening a detail panel from within
    another detail panel is a legitimate navigation pattern. Each subsequent sheet offsets
    inward, creating a visual stack that communicates depth without obscuring the panels
    behind it. The user can dismiss individual sheets or close the entire stack at once.
    This is horizontal depth, not vertical depth: the panels fan out like cards in a hand,
    not like windows piled on a desktop.
  </p>
</div>
